# RPG Maker Style Hub Expansion

- Version: 0.1
- Date: 2026-04-16
- Project id: `zunyang-party-panic`
- Based on: `docs/01-birthday-chat-game-plan.md`, `docs/08-prototype-scope-and-mvp-build-plan.md`, current `libGDX` prototype
- Goal of this document: 현재 `방송형 라운드 프로토타입`을 버리지 않고, `쯔꾸르풍 허브 + 미니게임 묶음` 구조로 확장하는 기준을 정리한다.

## 1. 이번 결정

이번 확장 방향은 아래 한 줄로 정리한다.

`전체 장르는 쯔꾸르풍 탐색/이벤트 게임으로 확장하고, 실제 플레이 코어는 미니게임 단위로 유지한다.`

중요:

- `실제 RPG Maker 엔진`으로 갈아타는 것은 아니다
- 현재 `Java + libGDX` 프로젝트를 유지한다
- 목표는 `쯔꾸르풍 연출과 구조`를 가져오는 것이다

즉, 엔진 교체가 아니라 `게임 구조 재설계`다.

## 2. 왜 이 방향이 맞는가

현재 프로토타입은 방송형 선택/사고/피날레 루프 자체는 괜찮다.  
문제는 이 상태로만 가면 `미니게임 모음`처럼 보이기 쉽다는 점이다.

쯔꾸르풍 구조를 얹으면 아래 장점이 생긴다.

- 캐릭터성과 세계관이 훨씬 잘 살아난다
- 생일 팬게임다운 `대화`, `방`, `오브젝트`, `이벤트`를 자연스럽게 넣을 수 있다
- 개별 미니게임이 전체 이야기 속 의미를 갖게 된다
- 결과가 바로 허브 공간에 반영되므로 성취감이 커진다
- 방송에서도 `방 탐색 -> 이벤트 -> 미니게임 -> 결과 리액션` 흐름이 더 잘 읽힌다

즉, `미니게임이 메인`이 아니라 `생일 하루를 진행하는 쯔꾸르풍 팬게임 안에 미니게임이 들어가는 구조`가 된다.

## 3. 새 장르 정의

### 장르 한 줄

`치즈냥의 생일 방송과 생일 파티를 준비하는 쯔꾸르풍 탐색 팬게임 + 방송형 미니게임`

### 플레이 판타지

플레이어는 치즈냥의 생일을 준비하는 공간 안을 돌아다니며,  
오브젝트와 대화하고, 미니게임을 열고, 선택 결과를 쌓아 마지막 생일 피날레를 완성한다.

### 게임 감정 흐름

아래 감정 흐름이 핵심이다.

1. 방에 들어와서 오늘의 분위기를 느낀다
2. 오브젝트를 눌러 대화/이벤트를 연다
3. 미니게임으로 생일 준비를 진행한다
4. 결과가 방 상태에 반영된다
5. 모든 준비가 끝나면 생일 피날레를 연다
6. 마지막에 엔딩 카드와 결과 대사가 나온다

## 4. 전체 구조

새 전체 구조는 아래처럼 잡는다.

```text
타이틀
  -> 허브 방 입장
  -> 오브젝트/NPC 상호작용
  -> 대화 또는 이벤트
  -> 해당 미니게임 진입
  -> 미니게임 결과 반영
  -> 허브로 복귀
  -> 다른 준비 진행
  -> 피날레 조건 충족
  -> 생일 피날레
  -> 엔딩 / 결과
```

핵심은 `허브 방`이 중심이라는 점이다.

현재 프로토타입의 선택/사고/피날레는 앞으로는 `한 개의 방송 준비 미니게임`이 되거나,  
혹은 `허브 안에서 열리는 여러 이벤트 중 하나`로 들어간다.

## 5. 권장 화면 구성

## 5-1. 타이틀 화면

역할:

- 게임 시작
- 이어하기 자리 확보
- 설정 / 종료

최소 구성:

- 시작
- 설정
- 종료

## 5-2. 허브 화면

이 화면이 사실상 메인이다.

역할:

- 방 안 이동 또는 포인트 선택
- 상호작용 가능한 오브젝트 확인
- 현재 생일 준비 진행도 확인
- 다음 해야 할 일 파악

허브 예시:

- 방송 책상
- 케이크 테이블
- 우편함
- 카메라 존
- 선물 더미
- 문 또는 피날레 스테이지

## 5-3. 대화/이벤트 화면

역할:

- 치즈냥 대사
- 팬레터 읽기
- 오늘 목표 설명
- 미니게임 진입 전 맥락 부여

쯔꾸르풍 감성은 사실 여기서 많이 나온다.

## 5-4. 미니게임 화면

역할:

- 현재 프로토타입처럼 즉시 플레이 중심
- 채팅 입력, 스트리머 개입, 점수 반영
- 짧고 방송적으로 읽히는 게임

미니게임은 독립 화면이어도 되고,  
허브 위에 이벤트 레이어처럼 띄워도 된다.

## 5-5. 결과 반영 화면

역할:

- 미니게임 성공/실패 표시
- 방 상태 변화
- 다음 이벤트 잠금 해제
- 치즈냥 리액션

## 5-6. 피날레 화면

역할:

- 모든 준비 결과 종합
- 최종 응원/클라이맥스
- 생일 축하 장면

## 5-7. 엔딩 화면

역할:

- 최종 등급
- 한 줄 멘트
- 오늘 완성된 파티 비주얼

## 6. 권장 상태 전이

현재의 단일 라운드 상태 머신은 앞으로 아래처럼 상위/하위 2층으로 나누는 것이 좋다.

### 상위 상태

- `BOOT`
- `TITLE`
- `HUB`
- `DIALOG`
- `MINIGAME`
- `FINALE`
- `ENDING`
- `SETTINGS`

### 하위 상태

`MINIGAME` 안에서는 게임별 상태를 따로 둔다.

예시:

- `INTRO`
- `COUNTDOWN`
- `ACTIVE`
- `RESOLVE`
- `RESULT`

즉, 지금 있는 `CHOICE / TROUBLE / FINALE` 구조는  
앞으로는 `특정 미니게임 내부 상태 머신`으로 내려가는 쪽이 맞다.

## 7. 첫 허브 기준 추천 구성

처음부터 큰 마을을 만들면 무조건 퍼진다.  
첫 버전 허브는 `한 방`으로 제한하는 것이 맞다.

### 추천 첫 허브

`치즈냥의 생일 방송 준비방`

구성 포인트:

- 중앙: 케이크 테이블
- 좌측: 방송 책상
- 우측: 포토존
- 하단: 팬레터 우편함
- 상단: 피날레 문 또는 무대

### 허브 안 상호작용 포인트 4개

1. 방송 책상
2. 팬레터 우편함
3. 포토존 카메라
4. 케이크 테이블

이 4개만으로도 허브 게임처럼 충분히 읽힌다.

## 8. 추천 미니게임 배치

현재 이미지와 기획 기준으로는 아래처럼 배치하면 자연스럽다.

### A. 방송 책상

- 미니게임 이름: `생일 방송 세팅`
- 현재 프로토타입과 가장 직접적으로 연결되는 슬롯
- 배경 투표, 방송 분위기 선택, 피날레 응원 같은 흐름을 여기로 넣기 좋다

현재 `PartyPanicScreen` 프로토타입은 이 슬롯의 첫 버전으로 보는 것이 가장 합리적이다.

### B. 팬레터 우편함

- 미니게임 이름: `팬레터 낭독`
- 선택형 또는 타이밍형 짧은 미니게임
- 감성 점수, 친밀도, 엔딩 멘트 강화에 반영

### C. 포토존 카메라

- 미니게임 이름: `생일 포토타임`
- 프레이밍 맞추기, 타이밍 맞추기, 채팅 반응 맞추기 같은 짧은 게임
- 지금 있는 `camera-chaos` 이벤트와 연결하기 좋다

### D. 케이크 테이블

- 미니게임 이름: `케이크 밸런스`
- 흔들리는 장식 정리, 순서 선택, 사고 수습형 게임
- 지금 있는 `cake-balance` 이벤트를 그대로 확장 가능

## 9. 현재 프로토타입을 어떻게 옮길 것인가

지금 만든 것은 버리는 게 아니라 `한 미니게임의 첫 프로토타입`으로 재정의하면 된다.

### 현재 프로토타입의 새 위치

현재 구조:

- 선택 라운드
- 사고 이벤트
- 피날레 응원

새 구조 해석:

- 이름: `방송 책상 미니게임`
- 의미: 오늘 생일 방송 세팅을 정하고 무사히 진행해 피날레까지 끌고 가는 게임

즉, 지금 코드는 `전체 게임`이 아니라 `허브 안 1개 콘텐츠`로 내려가면 된다.

- `world`: 저장 가능한 전체 진행도

## 11. 현재 코드 기준 최소 리팩토링 방향

큰 폭 리라이트보다 아래 순서가 맞다.

### 1단계

현재 `DesktopLauncher` 안 익명 `Game`을 분리해 `PartyPanicGame` 클래스로 올린다.

이유:

- 화면 전환을 시작하려면 `Game#setScreen(...)`를 직접 다루는 상위 앱 객체가 있어야 한다
- 지금처럼 한 화면에 다 몰아두면 허브/대화/미니게임 분리가 힘들다

### 2단계

현재 `PartyPanicScreen`을 `BroadcastMinigameScreen`으로 이름과 책임을 재정의한다.

이유:

- 지금 코드는 이미 하나의 게임 루프를 갖고 있다
- 앞으로 이 화면은 전체 게임이 아니라 `방송 책상 미니게임`이 된다

### 3단계

`HubScreen`을 만든다.

최소 기능:

- 배경 1장
- 상호작용 포인트 4개
- 키보드 또는 마우스로 선택
- 엔터 또는 클릭으로 진입

### 4단계

`GameProgress`를 추가한다.

최소 저장 값:

- 어떤 미니게임을 끝냈는지
- 각 미니게임 점수
- 허브 장식 상태
- 피날레 잠금 해제 여부
- 최종 결과 등급

### 5단계

미니게임 종료 후 허브로 복귀시키고, 방 상태를 실제로 바꾼다.

예시:

- 방송 책상 완료 -> 배경 조명 변화
- 팬레터 완료 -> 우편함 반짝임 제거
- 포토타임 완료 -> 포토존 장식 추가
- 케이크 완료 -> 케이크 비주얼 변화

## 12. 방송용 운영 관점

쯔꾸르풍으로 가도 방송형 설계는 유지해야 한다.

즉, 아래는 유지하는 편이 좋다.

- 좌측: 게임 본화면
- 우측: 스트리머 운영 패널
- 상단 또는 좌상단: 현재 목표
- 하단: 입력 힌트

허브 상태에서는 패널 내용만 바뀌면 된다.

예시:

- 현재 완료한 준비 항목
- 다음 추천 상호작용
- 스트리머 전용 강제 진행 버튼
- 채팅 참여 상태

## 13. 첫 MVP 컷

쯔꾸르풍 확장 첫 MVP는 아래 정도면 충분하다.

- 타이틀 화면 1개
- 허브 방 1개
- 상호작용 포인트 4개
- 실제 플레이 가능한 미니게임 2개
  - 방송 책상
  - 케이크 또는 포토타임
- 대화 이벤트 2~3개
- 피날레 1개
- 엔딩 결과 화면 1개

중요:

- 미니게임을 처음부터 4개 다 만들 필요는 없다
- 허브 구조가 먼저 살아야 한다
- 첫 MVP는 `허브 진입 -> 미니게임 2개 -> 피날레`만 되어도 충분히 쯔꾸르풍으로 읽힌다

## 14. 지금 기준 추천 구현 순서

아래 순서가 가장 현실적이다.

1. `PartyPanicGame` 상위 진입 객체 분리
2. `PartyPanicScreen` -> `BroadcastMinigameScreen` 역할 재정의
3. `HubScreen` 추가
4. `GameProgress` 추가
5. 허브에서 방송 책상 미니게임으로 진입 연결
6. 미니게임 종료 후 허브 복귀
7. 두 번째 미니게임 추가
8. 피날레 화면 추가
9. 엔딩 결과 화면 추가

## 15. 결론

이 프로젝트는 `미니게임 모음`보다 `쯔꾸르풍 생일 팬게임 안에 미니게임이 들어가는 구조`가 더 맞다.

현재 프로토타입은 실패한 방향이 아니라,  
앞으로 `허브 안에 들어갈 첫 번째 미니게임`으로 재배치하면 된다.

즉, 지금의 올바른 다음 문장은 아래다.

`현재 libGDX 프로토타입을 유지하되, 상위 구조를 쯔꾸르풍 허브/대화/피날레 흐름으로 확장하고, 기존 라운드형 플레이는 허브 안의 방송 책상 미니게임으로 재정의한다.`
