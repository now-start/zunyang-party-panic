이번 문서에서는 아래를 확정한다.

- 첫 프로토타입에서 반드시 들어갈 것
- 지금 당장 만들지 않을 것
- 어떤 순서로 구현할 것
- 무엇을 성공 기준으로 볼 것
- 실제 채팅 연동 전에 어떤 로컬 시뮬레이터가 필요한지

## 2. 현재 기준 결론

지금까지의 문서를 기준으로 보면, 첫 구현 목표는 `완성품`이 아니라 아래에 가깝다.

`스트리머가 직접 조작 가능하고, 적은 채팅량도 가정한 상태에서 한 판이 처음부터 끝까지 끊기지 않고 돌아가는 방송형 생일 팬게임 프로토타입`

즉, 현재 목표는 화려한 상용 수준이 아니라 `방송에서 실제로 굴릴 수 있는 구조 검증`이다.

## 3. 프로토타입과 MVP의 구분

### 프로토타입

가장 먼저 만들 버전이다.

목적:

- 게임 루프가 자연스러운지 확인
- 스트리머 개입 버튼이 실제로 재미를 만드는지 확인
- 저채팅량 fallback이 어색하지 않은지 확인
- 한 화면 구조가 방송용으로 읽히는지 확인

### MVP

프로토타입 검증 후 실제 방송 테스트 가능한 최소 버전이다.

목적:

- 한 판을 안정적으로 반복 진행
- 시청자 참여 흐름 검증
- 방송 오퍼레이션 검증

즉, 프로토타입은 `내부 검증`, MVP는 `방송 투입 가능 최소선`이다.

## 4. 첫 프로토타입의 성공 기준

첫 프로토타입은 아래를 만족하면 성공으로 본다.

### 기능 기준

- 한 판이 `IDLE -> COUNTDOWN -> CHOICE -> TROUBLE -> FINALE -> RESULT -> IDLE`로 정상 순환
- 스트리머 버튼이 상태에 맞게 활성/비활성
- 무응답/동률/fallback이 작동
- 선택 결과, 사고 결과, 피날레 결과가 화면에 반영

### 운영 기준

- 스트리머 혼자서도 흐름을 이해하고 조작 가능
- 채팅 시뮬레이터로 적은 입력만 넣어도 판이 멈추지 않음
- 결과 화면이 방송적으로 읽힘

### 방송 기준

- OBS에서 좌측 방송 영역만 잘라 써도 정보가 충분함
- 우측 조작 패널이 없어도 시청자는 현재 상황을 이해할 수 있음

## 5. 프로토타입 포함 범위

첫 프로토타입에 반드시 포함할 항목은 아래와 같다.

### A. 기본 창 구조

- `1920x1080` 기준 단일 창
- 좌측 방송 영역
- 우측 스트리머 조작 패널
- 상단 HUD
- 하단 명령어 바

### B. 전체 상태 머신

- `07` 문서 기준 핵심 상태 흐름
- 상태 전이에 따른 화면 갱신
- 버튼 활성 조건 반영

### C. 콘텐츠 최소 세트

- 선택 라운드 2개
  - 케이크 베이스
  - 장식 아이템
- 사고 이벤트 3개
  - 돌풍
  - 고양이 습격
  - 선물 낙하
- 결과 등급 5개
  - S / A / B / C / D

### D. 스트리머 액션 최소 세트

- Start
- Today's Pick
- Reroll
- Birthday Wish
- Finale Trigger
- Result Comment

### E. 로컬 채팅 시뮬레이터

실제 플랫폼 채팅 연동 없이도 아래가 가능해야 한다.

- 가짜 유저 8명~15명 입력
- 입력 속도 조절
- 특정 명령어 주입
- 무응답 상황 재현
- 동률 상황 재현

이 시뮬레이터는 사실상 필수다.

## 6. 프로토타입 제외 범위

첫 프로토타입에서 빼야 하는 항목은 아래와 같다.

### 시스템 측면

- 실제 CHZZK 채팅 연동
- 계정 인증
- 사용자 데이터 저장
- 방송 기록 저장
- 통계 대시보드

### 비주얼 측면

- 최종 일러스트 품질
- 고급 파티클
- 정교한 애니메이션
- 완성형 사운드 디자인

### 콘텐츠 측면

- 다수의 선택지 세트
- 다수의 테마 스킨
- 이벤트 분기 대량 추가
- 확장형 엔딩 컬렉션

첫 프로토타입은 `구조 검증`이 목적이므로, 볼륨을 욕심내면 안 된다.

## 7. MVP 포함 범위

프로토타입 이후 MVP에서는 아래 정도까지 확장하면 된다.

- 실제 방송 화면 수준으로 다듬은 UI
- 1판 반복 운용 안정화
- 2~3개의 선택 세트 추가
- 1~2개의 사고 이벤트 추가
- 피날레 연출 강화
- 스트리머 개입 피드백 시각 강화
- 실제 채팅 연동 검토 또는 테스트 어댑터

즉, MVP는 프로토타입을 `방송 투입 가능 수준`으로 끌어올리는 단계다.

## 8. 권장 구현 순서

아래 순서가 가장 안전하다.

### Step 1. 빈 화면 뼈대

- 좌측 방송 영역
- 우측 조작 패널
- 상단 HUD
- 하단 명령어 바

목적:

- 레이아웃 고정
- OBS 기준 안전 구역 확인

### Step 2. 상태 머신만 구현

- 내용 없는 더미 상태라도 전이되게 만들기
- COUNTDOWN -> CHOICE -> TROUBLE -> FINALE -> RESULT 순환

목적:

- 흐름이 먼저 끊기지 않는지 확인

### Step 3. 스트리머 버튼 연결

- Start
- Pick
- Reroll
- Wish
- Finale

목적:

- 스트리머가 실제로 판을 굴리는 감각 확인

### Step 4. 채팅 시뮬레이터 연결

- 가짜 유저 입력 반영
- 동률/무응답 재현

목적:

- 저채팅량 운영 검증

### Step 5. 실제 콘텐츠 연결

- 케이크 베이스
- 장식 아이템
- 돌풍 / 고양이 / 선물
- 결과 등급

목적:

- 게임처럼 보이기 시작하는 최소선 달성

### Step 6. fallback 및 결과 연출 보강

- 무응답 처리
- 소박한 피날레
- 스트리머 판정 배너

목적:

- 방송 중 민망한 빈 구간 방지

## 9. 마일스톤 제안

### M1. Clickable Prototype

정의:

- 화면 전환 가능
- 버튼 반응 가능
- 채팅 없이도 한 판 진행 가능

### M2. Playable Prototype

정의:

- 시뮬레이터 입력으로 한 판 정상 진행
- 결과 등급과 피날레 작동
- fallback 작동

### M3. Broadcast Rehearsal Build

정의:

- OBS 기준 화면 구조 확인
- 스트리머 1인 리허설 가능
- 3판 연속 진행 가능

## 10. 로컬 시뮬레이터 요구사항

이 프로젝트는 실제 채팅 연동보다 먼저 시뮬레이터가 필요하다.

### 최소 기능

- 유저명 기반 가짜 입력
- `1`, `2`, `3`, `!축하`, `!응원` 입력
- 속도 프리셋
  - 느림
  - 보통
  - 혼잡
- 무응답 모드
- 동률 유도 모드

### 있으면 좋은 기능

- 특정 선택지 편향 입력
- 특정 유저 반복 입력 제한 테스트
- 피날레 참여자 수 조절

## 11. 검증 체크리스트

### 흐름 체크

- 상태가 꼬이지 않는가
- 스트리머가 다음 행동을 쉽게 이해하는가
- 화면만 봐도 시청자가 현재 해야 할 행동을 이해하는가

### 콘텐츠 체크

- 선택지가 즉시 읽히는가
- 결과 차이가 눈에 띄는가
- 사고 대응이 지나치게 빠르거나 비지 않는가

### 운영 체크

- 채팅 5명 참여만 있어도 한 판이 성립하는가
- 무응답 시 어색하지 않은가
- 동률 시 스트리머 판정이 자연스러운가

### 방송 체크

- OBS 크롭이 쉬운가
- 우측 패널 없이도 방송 화면이 완결되는가
- 결과 화면이 클립 포인트로 충분한가

## 12. 주요 리스크

### 1. 구조는 좋은데 실제로는 심심할 수 있음

대응:

- 결과 반영 속도 강화
- 스트리머 개입 피드백 강화

### 2. 저채팅량에서 피날레가 자주 약해질 수 있음

대응:

- 참여자 수 기준 완화
- 소규모 축하 엔딩 보강

### 3. 스트리머 조작이 많아질 수 있음

대응:

- 프로토타입 단계에서는 버튼 수를 고정
- 긴급 버튼 외에는 자동 진행 유지

## 13. 지금 기준 권장 MVP 컷

현재 시점에서 "이 정도면 첫 구현 착수 가능"한 컷은 아래다.

- 한 판 전체 루프 동작
- 2개 선택 라운드
- 3개 사고 이벤트
- 5개 스트리머 버튼
- 로컬 채팅 시뮬레이터
- 결과 등급과 결과 카드
- 저채팅량 fallback

이 이상부터는 2차 작업으로 넘기는 편이 맞다.
