# Birthday Game Loop Design

- Version: 0.1
- Date: 2026-04-16
- Based on: `docs/01-birthday-chat-game-plan.md`, `docs/02-birthday-game-genre-direction.md`
- Goal of this document: 한 판의 전체 흐름, 라운드 구조, 승패 조건, 채팅 입력 타이밍, 방송 화면 표시 요소를 구체화한다.

## 1. 이번 단계에서 확정할 것

이번 문서는 실제 방송에서 한 판이 어떻게 진행되는지를 정하는 문서다.

이번 단계에서 확정할 항목은 아래와 같다.

- 한 판의 총 길이
- 라운드 수와 각 라운드의 역할
- 채팅 입력 방식과 집계 타이밍
- 점수 구조와 결과 등급
- 방송 화면에서 꼭 보여야 하는 정보

## 2. 한 판의 목표 길이

한 판의 권장 길이는 `100초 ~ 140초`로 잡는다.

이 범위가 적절한 이유는 아래와 같다.

- 룰을 기억한 상태에서 바로 다시 돌리기 좋다
- 방송 템포가 늘어지지 않는다
- 채팅 참여 피로가 크지 않다
- 1회성 이벤트가 아니라 여러 판 반복하기 좋다

권장 기준은 아래와 같다.

- 짧은 판: 약 100초
- 표준 판: 약 120초
- 긴 판: 약 140초

초기 기준은 `표준 판 120초`를 추천한다.

다만 이 기준은 `평시 채팅량이 낮지 않은 경우`에 더 잘 맞는다.  
현재 운영 가정이 `분당 채팅 10개 내외`, `고유 시청자 40명 내외`라면, 실제 게임 설계는 `저채팅량 대응형`으로 보정해야 한다.

즉, 실제 MVP 운영 기준은 아래처럼 조정하는 편이 적절하다.

- 표준 판: 약 130초 ~ 160초
- 선택 입력 구간: 조금 더 길게
- 사고 대응 판정: 2초 단위 연타형이 아니라 5초 이상 단위의 묵직한 판정형
- 피날레 성공 기준: 폭발적 채팅량보다 `참여자 수` 중심

## 3. 한 판의 전체 구조

한 판은 총 5개 구간으로 나눈다.

1. 라운드 시작 안내
2. 파티 준비 선택
3. 파티 사고 대응
4. 마지막 축하 피날레
5. 결과 공개

이 구조는 `선택`, `반응`, `집단 폭발`, `결과 리액션`이 모두 들어가도록 만든다.

추가로 이 게임은 각 구간마다 `스트리머 개입 포인트`가 있어야 한다.

- 채팅은 주된 결정권을 가진다
- 스트리머는 판을 연출하고 결과를 흔드는 제한적 개입권을 가진다
- 즉, 자동 집계만 돌아가는 구조가 아니라 `채팅 주도 + 스트리머 연출` 구조여야 한다

## 4. 표준 판 상세 루프

### Phase 0. 오프닝 카운트다운

- 길이: 5초
- 목적: 이번 판 시작을 명확히 알림
- 화면 내용:
  - 게임 타이틀
  - "곧 생일 파티 준비 시작"
  - 카운트다운 숫자

채팅 입력은 이 단계에서 받지 않는다.

스트리머 개입:

- 시작 버튼으로 직접 판 개시
- 짧은 오프닝 멘트
- 이번 판 분위기를 정하는 한 줄 선언

### Phase 1. 파티 준비 선택 1

- 길이: 18초
- 목적: 첫 준비 요소 확정
- 예시 주제:
  - 케이크 시트
  - 메인 컬러
  - 파티 테마

채팅 입력 방식:

- 입력 키: `1`, `2`, `3`
- 집계 방식: 같은 유저는 마지막 입력 1회만 반영
- 종료 시 최다 득표 선택지 확정

화면 연출:

- 세 가지 선택지가 크게 표시
- 각 선택지 위에 실시간 득표 바 표시
- 남은 시간 표시

스트리머 개입:

- 세 선택지 중 하나를 `오늘의 픽`으로 지정 가능
- `오늘의 픽`은 채팅이 고르면 추가 준비도 보너스 획득
- 단, 최종 선택 자체를 강제로 바꾸지는 않음

### Phase 2. 파티 준비 선택 2

- 길이: 18초
- 목적: 두 번째 준비 요소 확정
- 예시 주제:
  - 장식 아이템
  - 초 종류
  - 선물 포장 방식

채팅 입력 방식:

- 입력 키: `1`, `2`, `3`
- 집계 방식: Phase 1과 동일
- 종료 시 최다 득표 선택지 확정

화면 연출:

- 선택 결과가 바로 배경 또는 오브젝트에 반영
- 누적된 파티 상태를 한눈에 보여 주기

스트리머 개입:

- 한 번에 한해 `선택지 새로고침` 가능
- 마음에 들지 않는 보기 구성이면 1회 리롤
- 이 기능은 판당 1회만 허용하는 것이 적절함

### Phase 3. 파티 사고 대응

- 길이: 18초 ~ 24초
- 목적: 생일 파티를 망치려는 사고를 막음
- 예시 사고:
  - 바람이 촛불을 끄려 함
  - 고양이가 케이크를 건드리려 함
  - 풍선이 터질 위기

채팅 입력 방식:

- 입력 키: `1`, `2`, `3`
- 의미: 좌 / 중 / 우 또는 사고 대응 대상 위치
- 집계 단위: 5초 ~ 6초
- 판정 횟수: 총 3회 정도
- 같은 유저는 집계 구간당 마지막 입력 1회만 반영

화면 연출:

- 중앙에 현재 사고가 크게 표시
- 채팅 다수 방향에 실시간 방어 효과 표시
- 성공 시 파티 안정도 증가
- 실패 시 파티 안정도 감소

이 단계는 `짧은 실시간 대응 미니액션` 구간이다.

단, 현재 예상 채팅량에서는 `초고속 반응형 미니액션`보다 `짧은 다중 위기 선택형`에 가깝게 설계하는 것이 안전하다.

스트리머 개입:

- 판당 1~2개의 `액티브 스킬` 사용 가능
- 예시:
  - `생일 소원`: 이번 판정 1회 자동 성공 보정
  - `장난 금지`: 3초 동안 사고 속도 완화
  - `촛불 보호`: 특정 위치를 잠깐 보호
- 채팅은 계속 입력하고, 스트리머는 위기 타이밍에 개입해 흐름을 뒤튼다

### Phase 4. 마지막 축하 피날레

- 길이: 18초 ~ 22초
- 목적: 전체 채팅이 하나로 몰리는 축하 장면 만들기

채팅 입력 방식:

- 기본 입력: `!축하`
- 선택 입력: `!응원`
- 집계 기준:
  - 총 입력 수
  - 고유 참여자 수
  - 연속 입력 유지량

권장 처리 방식:

- 같은 유저는 2초 쿨다운
- 도배성 입력은 제한
- 고유 참여자 수 비중을 크게 둠

화면 연출:

- 화면 전체 게이지 상승
- 배경 이펙트 점점 강화
- 마지막 3초에는 대형 텍스트와 효과음으로 몰입 극대화

스트리머 개입:

- 축하 게이지가 임계치에 도달하면 `피날레 트리거` 버튼 활성화
- 스트리머가 직접 버튼을 눌러 폭죽, 컷신, 대형 축하 연출 발동
- 게이지를 채우는 것은 채팅, 축하를 완성시키는 것은 스트리머라는 역할 분리가 중요함

### Phase 5. 결과 공개

- 길이: 10초 ~ 15초
- 목적: 한 판의 결과를 즉시 보상처럼 보여 줌

결과 화면에 포함할 정보:

- 최종 등급
- 완성된 파티 이름
- 가장 많이 선택된 요소
- 총 참여자 수
- 총 채팅 입력 수
- 한 줄 결과 멘트

예시 한 줄 결과 멘트:

- "완벽한 생일 파티였어요"
- "조금 난장판이었지만 즐거웠어요"
- "파티는 터졌지만 축하는 성공했어요"

스트리머 개입:

- 결과 공개 직후 `베스트 순간` 또는 `오늘의 문제아`를 한 줄로 직접 선택 가능
- 결과 화면에 스트리머 코멘트가 함께 뜨면 방송 컨텐츠로서 밀도가 올라감

## 5. 라운드별 감정 곡선

한 판의 감정 흐름은 아래처럼 잡는 것이 좋다.

1. 시작 기대감
2. 준비 단계의 선택 재미
3. 사고 대응 단계의 긴장감
4. 마지막 축하 단계의 폭발감
5. 결과 공개의 웃음 또는 감동

이 흐름이 있어야 단순한 투표 게임이 아니라 방송용 게임처럼 느껴진다.

## 6. 점수 구조

이번 게임은 단순 합산 점수보다 `파티 상태`를 관리하는 방식이 잘 맞는다.

권장 스탯은 아래 3개다.

- `준비도`: 선택 라운드에서 올라감
- `안정도`: 사고 대응 라운드에서 오르거나 내려감
- `축하열기`: 마지막 피날레에서 올라감

각 스탯은 0~100 범위로 관리하는 방향이 적절하다.

### 준비도

- 좋은 조합 선택 시 상승
- 평범한 선택 시 소폭 상승
- 웃긴 꽝 조합 선택 시 낮게 상승하거나 약간 감소

### 안정도

- 사고 대응 성공 시 상승 또는 유지
- 대응 실패 시 감소
- 완전 실패해도 즉시 게임 오버는 아니다

### 축하열기

- 고유 참여자 수에 크게 영향 받음
- 총 입력 수는 보조 점수로 반영
- 마지막 판정 비중을 조금 더 높여도 좋음

## 7. 승리와 실패 구조

이 게임은 `완전 실패로 끝나는 게임`보다 `등급이 달라지는 게임`이 더 적합하다.

즉, 기본 구조는 아래와 같다.

- 게임 오버보다 결과 등급 차등
- 실패해도 결과 화면은 반드시 보여 줌
- 낮은 등급도 웃긴 리액션으로 소비 가능해야 함

### 권장 등급 구조

- `S`: 완벽한 생일 파티
- `A`: 대성공 파티
- `B`: 성공적인 파티
- `C`: 정신없는 파티
- `D`: 대환장 파티

### 등급 판정 예시

- 준비도 35%
- 안정도 25%
- 축하열기 40%

생일 팬게임 특성상 마지막 축하 구간 비중을 가장 높게 잡는 편이 맞다.

## 8. 채팅 입력 타이밍 규칙

채팅 시스템은 단순해야 하지만, 판정 규칙은 명확해야 한다.

### 공통 원칙

- 각 라운드에서 현재 허용되는 입력을 화면 하단에 항상 표시
- 허용되지 않은 입력은 무시
- 같은 유저는 집계 구간 안에서 마지막 입력만 인정

### 선택 라운드

- 허용 입력: `1`, `2`, `3`
- 집계 단위: 라운드 종료 시 일괄 집계
- 표시 방식: 실시간 비율 표시 가능

### 사고 대응 라운드

- 허용 입력: `1`, `2`, `3`
- 집계 단위: 5초 ~ 6초마다 판정
- 표시 방식: 현재 다수 방향 강조

### 축하 피날레

- 허용 입력: `!축하`, `!응원`
- 집계 단위: 실시간 누적
- 표시 방식: 전체 축하 게이지와 화면 효과 증가

## 9. 저채팅량 운영 기준

현재 예상 운영값은 아래처럼 둔다.

- 평시 채팅량: 분당 약 10개
- 고유 시청자 수: 약 40명
- 실제 게임 활성 참여자: 약 8명 ~ 15명 예상

이 기준이면 설계 원칙은 아래처럼 바뀌어야 한다.

### 원칙 1. 적은 수의 입력으로도 의미 있는 결과가 나와야 한다

한 라운드에서 채팅 수십 개를 기대하면 안 된다.  
5명만 반응해도 판이 굴러가게 해야 한다.

### 원칙 2. 속도보다 명확성이 중요하다

2초 반응형은 비어 보일 위험이 크다.  
대신 5초 이상 동안 채팅이 보고 판단하고 입력할 시간을 줘야 한다.

### 원칙 3. 승패보다 참여 흔적을 살려야 한다

절대치가 낮더라도 "이번엔 9명이 같이 밀어줬다" 같은 식으로 의미를 부여해야 한다.

### 원칙 4. 스트리머 개입 비중을 약간 올려야 한다

채팅이 적게 들어오는 구간은 스트리머 개입과 리액션으로 밀도를 보강해야 한다.

## 10. 방송 화면 정보 설계

Windows 클라이언트 기준으로 한 화면에 아래 정보가 보여야 한다.

### 필수 표시 요소

- 현재 라운드 이름
- 남은 시간
- 현재 입력 가능한 채팅 명령어
- 중앙 메인 파티 화면
- 준비도 / 안정도 / 축하열기 상태
- 현재 선택 결과 또는 위험 상황
- 현재 사용 가능한 스트리머 액션 버튼
- 스트리머가 이미 사용한 개입 횟수 또는 스킬 상태

### 결과 화면 필수 요소

- 최종 등급
- 완성된 파티 비주얼
- 총 참여 인원
- 총 채팅 수
- 스트리머가 읽기 좋은 결과 한 줄

## 11. 스트리머 진행 기준 메모

이 게임은 스트리머가 중간중간 멘트를 얹기 쉬워야 한다.

권장 멘트 포인트는 아래와 같다.

- 시작: "이번엔 제대로 된 생일 파티 가자"
- 선택 라운드: "이걸 고른다고?"
- 사고 대응: "야야야 그쪽 막아!"
- 피날레: "다 같이 축하 눌러!"
- 결과: "이게 성공이 맞아?"

그리고 멘트만으로 끝나면 부족하다.  
실제로는 아래처럼 시스템 개입이 함께 있어야 한다.

- 판 시작
- 오늘의 픽 지정
- 선택지 리롤
- 위기 대응 스킬 사용
- 피날레 트리거 발동
- 결과 코멘트 선택

즉, 시스템 자체가 진행을 대신하는 것이 아니라, `스트리머가 판을 돌리는 감각`을 받쳐 주는 구조여야 한다.

## 12. 한 판 요약 시나리오

표준 한 판은 아래처럼 흘러간다.

1. 카운트다운 시작
2. 채팅이 케이크와 장식을 정함
3. 중간에 사고가 나고 채팅이 급하게 막음
4. 마지막에 모두가 `!축하`를 치며 게이지를 채움
5. 파티 결과가 등급과 함께 공개됨

이 요약이 곧 게임의 핵심 방송 경험이다.

## 13. 이번 단계 결론

현재 기준 한 판 구조는 아래 문장으로 정리할 수 있다.

`한 판은 약 120초 분량으로, 두 번의 준비 선택 라운드와 한 번의 실시간 사고 대응 라운드, 마지막 집단 축하 피날레, 그리고 결과 공개로 구성하며, 각 구간마다 스트리머가 제한된 시스템 개입권을 가진다.`

## 14. 다음 문서에서 다룰 내용

다음 단계에서는 아래를 정리하면 된다.

1. 각 라운드에서 실제로 어떤 선택지가 나오는지
2. 파티 결과가 어떤 조합으로 달라지는지
3. 어떤 사고 이벤트를 몇 종 넣을지
4. 결과 멘트와 엔딩 연출 톤을 어떻게 나눌지

즉, 다음 문서는 `콘텐츠 테이블 설계`가 맞다.
